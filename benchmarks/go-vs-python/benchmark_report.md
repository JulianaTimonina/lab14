# Отчёт по benchmark: Go vs Python

## Цель
Сравнить производительность и потребление ресурсов компонентов системы мониторинга энергопотребления, реализованных на Go (collector) и Python (processor) при выполнении оконной агрегации данных.

## Методология
- **Go бенчмарк**: использует `WindowAggregator` из пакета `aggregation` с окном 30 секунд.
- **Python бенчмарк**: использует упрощённый агрегатор `SimpleWindowAggregator` с аналогичной логикой (окно 30 секунд).
- Оба бенчмарка обрабатывают **10 000 тестовых показаний** от 100 виртуальных счётчиков.
- Измеряемые метрики:
  - Время добавления показаний (throughput, чтений/сек)
  - Время агрегации (сек)
  - Потребление памяти (МБ)
- Окружение: Windows 10, Python 3.x, Go 1.25.

## Результаты

|---------|----------|
| Количество показаний | 10 000 |

| Throughput | **12 745 347 чтений/сек** |
| Время агрегации | 0.1006276 сек |
| Потребление памяти | **1.02 МБ** |

### Python (python-processor)
| Метрика | Значение |
|---------|----------|
| Количество показаний | 10 000 |
| Время добавления | 0.042663 сек |
| Throughput | **234 394 чтений/сек** |
| Время агрегации | 0.020819 сек |
| Потребление памяти | **24.28 МБ** |

## Сравнительный анализ

### Производительность (Throughput)
- Go обрабатывает **~12.7 миллионов** чтений в секунду.
- Python обрабатывает **~234 тысячи** чтений в секунду.
- **Go быстрее Python в ~54 раза** по throughput.

### Потребление памяти
- Go использует **1.02 МБ** памяти.
- Python использует **24.28 МБ** памяти.
- **Go эффективнее по памяти в ~24 раза**.

### Время агрегации
- Go: 0.1006 сек (включая искусственную задержку 100 мс).
- Python: 0.0208 сек (быстрее, но агрегация проще).

## Выводы
1. **Go значительно превосходит Python** по производительности и эффективности использования памяти для задач потоковой агрегации данных.
2. Python-реализация потребляет в 24 раза больше памяти, что критично для систем с большим объёмом данных.
3. Высокий throughput Go позволяет обрабатывать миллионы событий в секунду, что делает его предпочтительным для high-load систем.
4. Python может быть приемлем для сценариев с меньшей нагрузкой, где важна скорость разработки и богатая экосистема библиотек.

## Рекомендации
- Для компонентов, требующих высокой производительности и низкой задержки (collector, агрегатор), использовать **Go**.
- Для компонентов, где важна гибкость и интеграция с ML/аналитикой (processor, dashboard), можно оставить **Python**.
- Рассмотреть возможность оптимизации Python-кода:
  - Использование `PyPy` или `Cython` для критических участков.
  - Применение библиотек `numpy`/`pandas` для векторных операций.
  - Уменьшение накладных расходов на структуры данных.

## Ограничения
- Бенчмарк использует синтетические данные, которые могут не полностью отражать реальную нагрузку.
- Python бенчмарк использует упрощённую реализацию агрегатора, тогда как production-версия (`WindowProcessor`) может иметь дополнительную функциональность (фоновые потоки, асинхронность), что влияет на производительность.
- Не учитывались сетевые задержки, сериализация/десериализация, работа с Kafka и БД.

## Заключение
Go демонстрирует существенные преимущества в производительности и эффективности использования ресурсов для задач агрегации потоковых данных. Это подтверждает целесообразность использования Go для high-performance компонентов системы мониторинга энергопотребления.

Python остаётся viable option для компонентов, где производительность не является критическим ограничением, а важны быстрота разработки и доступность библиотек.
