# План бенчмарков: Сравнение производительности Go vs Python

## Цель
Сравнить производительность Go- и Python-версий сборщика данных по следующим метрикам:
1. Пропускная способность (записей/сек)
2. Задержка обработки (мс)
3. Потребление памяти (MB)
4. Использование CPU (%)
5. Объём передаваемых данных (MB/сек)

## Тестовые сценарии

### Сценарий 1: Сбор сырых данных
- **Нагрузка**: 100 счётчиков, опрос каждые 10 секунд
- **Длительность**: 5 минут
- **Метрики**: 
  - Количество успешно собранных записей
  - Среднее время ответа эмулятора
  - Потребление памяти сборщиком

### Сценарий 2: Оконная агрегация
- **Нагрузка**: 1000 записей/сек
- **Окно**: 30 секунд (tumbling window)
- **Агрегации**: сумма, среднее, минимум, максимум
- **Метрики**:
  - Время агрегации окна
  - Задержка между получением данных и отправкой агрегатов
  - Потребление памяти во время агрегации

### Сценарий 3: Передача данных
- **Форматы**: JSON vs Apache Arrow
- **Объём данных**: 10,000 записей
- **Метрики**:
  - Размер данных в байтах
  - Время сериализации/десериализации
  - Время передачи по сети
  - Пропускная способность

### Сценарий 4: Валидация данных
- **Библиотеки**: Go (нативная) vs Rust (через cgo) vs Python (PyO3)
- **Нагрузка**: 100,000 записей
- **Метрики**:
  - Время валидации
  - Потребление памяти
  - Количество false positives/negatives

### Сценарий 5: Интеграционный тест
- **Полный конвейер**: сбор → валидация → агрегация → передача
- **Нагрузка**: 100 счётчиков, 10 записей/сек на счётчик
- **Длительность**: 10 минут
- **Метрики**:
  - End-to-end задержка
  - Общая пропускная способность
  - Стабильность работы (память, CPU)

## Инструменты для бенчмарков

### Для Go:
- Встроенный `testing.B` для микро-бенчмарков
- `pprof` для профилирования CPU и памяти
- `go test -bench` для запуска бенчмарков
- Prometheus для сбора метрик в реальном времени

### Для Python:
- `timeit` для измерения времени выполнения
- `memory_profiler` для профилирования памяти
- `pytest-benchmark` для структурированных бенчмарков
- `psutil` для мониторинга системных ресурсов

### Общие:
- Docker для изоляции окружения
- Grafana для визуализации результатов
- Jupyter Notebook для анализа данных

## Этапы выполнения

### Этап 1: Подготовка (1 день)
1. Настройка тестового окружения

3. Настройка мониторинга (Prometheus + Grafana)
4. Подготовка скриптов запуска

### Этап 2: Микро-бенчмарки (2 дня)
1. Тестирование отдельных компонентов
2. Измерение базовой производительности
3. Оптимизация критических участков
4. Документирование результатов

### Этап 3: Интеграционные тесты (2 дня)
1. Запуск полного конвейера
2. Измерение end-to-end метрик
3. Стресс-тестирование
4. Анализ узких мест

### Этап 4: Анализ и отчёт (1 день)
1. Сравнение результатов
2. Выявление преимуществ и недостатков
3. Формулирование рекомендаций
4. Подготовка финального отчёта

## Ожидаемые результаты

### Go (ожидаемые преимущества):
- Высокая пропускная способность (> 10,000 записей/сек)
- Низкое потребление памяти (< 100 MB)
- Предсказуемая задержка
- Эффективная работа с горутинами

### Python (ожидаемые преимущества):
- Быстрая разработка и прототипирование
- Богатая экосистема библиотек
- Удобство интеграции с ML/аналитикой
- Гибкость конфигурации

### Критерии выбора:
1. **Высокая нагрузка** → Go
2. **Сложная обработка данных** → Python
3. **Реальное время** → Go
4. **Быстрое прототипирование** → Python
5. **Ресурсоограниченная среда** → Go

## Риски и ограничения

### Риски:
1. Недостаточная нагрузка для выявления различий
2. Проблемы с настройкой окружения
3. Искажение результатов из-за фоновых процессов
4. Сложность воспроизведения результатов

### Ограничения:
1. Тестирование только на конкретном железе
2. Ограниченное время на тестирование
3. Упрощённые сценарии по сравнению с production
4. Отсутствие тестирования отказоустойчивости
